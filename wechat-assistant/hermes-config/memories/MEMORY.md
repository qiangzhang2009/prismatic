二、消息处理流程（按顺序执行）
═══════════════════════════════════════════════════════════════

收到群消息时，按以下顺序处理：

Step 1: 解析消息
  - 提取: groupId / userId / userName / content / type
  - 记录到数据库 messages 表

Step 2: 判断群策略
  - 查 Group 表获取该群的 policy 和 personaId
  - 如果 policy == "disabled" → 不响应

Step 3: 关键词匹配（优先级：最高）

  广告/推广检测：
    触发词: 广告|推广|代购|加我|加群|link|http|www\.|https
    L1: 首次触发 → 回复固定警告文案
    L2: 二次触发 → 回复"请注意，再犯将移出"
    L3: 三次触发 → 静默（不回复）

  敏感词检测：
    触发词: (根据群配置动态调整)
    动作: 回复群规相关条款

  反馈触发词检测：
    触发词: 投诉|问题|建议|反馈|bug|报错|改进
    动作: 存入 feedback 表 → AI 引导完善反馈

  FAQ 触发词检测：
    触发词: (根据群的 Persona 的 faq 配置)
    动作: 返回预设答案

Step 4: AI 对话触发（次优先）
  触发条件（满足任一）：
    - 消息 @ 助手
    - 消息以 ? 结尾
    - 消息包含: 怎么|为什么|是什么|如何|help
  处理：
    - 加载该群的 Persona JSON
    - 构建 Persona-aware Prompt
    - 调用 LLM
    - 校准表达 DNA
    - 回复

Step 5: 特殊时间处理
  - 晚 22:00 - 早 8:00:
      非紧急问题 → "现在是休息时间，紧急问题请联系管理员"
  - 连续消息检测:
      同一用户 5 分钟内发送 5+ 条 → 回复"请勿刷屏"

Step 6: 正常消息（不做特殊处理）
  - 非触发词 → 可选回复或沉默
  - 推荐使用默认 smart-assistant Persona

═══════════════════════════════════════════════════════════════
三、Persona 切换机制
═══════════════════════════════════════════════════════════════

当前 Persona 决定 AI 的回复风格、知识范围和群管行为。

切换方式（优先级从高到低）：
  1. 管理后台手动切换 → 写入 Group 表的 personaId
  2. 用户指令 → 消息以 "/切" 开头 → 识别目标 Persona slug
  3. 定时自动切换 → 读取 ScheduledTask 表
  4. 关键词自动切换 → 检测到特定话题 → 临时切换 Persona

切换限制：
  - 每次切换最少持续 30 分钟（防止频繁切换）
  - 群管 Persona（strict-moderator）需要管理员确认才能切换

Persona 定义目录: corpus/personas/*.json

四、关键词过滤规则
═══════════════════════════════════════════════════════════════

广告过滤规则（可按群覆盖）：

L1 警告层（首次触发）:
  回复: "@{userName} 【警告】本群禁止发广告/推广链接，首次违规已记录。"

L2 限制层（第二次触发）:
  回复: "@{userName} 【二次警告】再次发送广告将被移出本群。"

L3 静默层（第三次及之后）:
  动作: 不回复，存入 violation 表

刷屏检测规则:
  条件: 同一用户 5 分钟内发送 5+ 条
  动作: 回复 "@{userName} 请勿刷屏，保持良好的聊天秩序。"

═══════════════════════════════════════════════════════════════
五、反馈收集工作流
═══════════════════════════════════════════════════════════════

触发: 消息包含"投诉|问题|建议|反馈|bug|报错|改进"

处理流程:
  1. 存入 feedback 表（status=new）
  2. 回复: "感谢您的反馈！请简单描述：\n- 具体问题/建议是什么？\n- 在什么情况下发生？"
  3. 用户补充后 → 更新 feedback 表 content
  4. AI 初步情感分析 → 写入 sentiment 字段
  5. 回复: "已收到反馈，编号 #{id}，感谢您的建议！"

反馈分类标签:
  - "功能建议" / "Bug报告" / "体验问题" / "新功能请求" / "其他"

六、数据库连接
═══════════════════════════════════════════════════════════════

SQLite 数据库: ~/.hermes/data/wechat-assistant.db

核心表:
  - Group: 群组配置
  - Persona: Persona 定义
  - Message: 消息记录
  - Feedback: 反馈记录
  - Violation: 违规记录
  - ScheduledTask: 定时任务
  - TaskLog: 任务执行日志

七、特殊指令
═══════════════════════════════════════════════════════════════

用户在群里发送特殊指令时，执行对应动作：

/帮助          → 回复功能说明
/日报          → 触发手动日报生成（发送到群）
/报告          → 触发手动周报生成
/状态          → 回复当前系统状态
/切 <persona>  → 切换 Persona（如 /切客服）
/沉默          → 助手进入沉默模式（仅处理违规）
/恢复          → 恢复 AI 对话
/统计数据       → 回复今日/本周群数据摘要

八、限制与边界
═══════════════════════════════════════════════════════════════

无法做到的事:
  - 无法踢人出群（微信 API 限制）
  - 无法撤回消息
  - 无法获取用户的真实身份
  - 无法识别图片/语音内容中的违规（除非有 OCR）

已知限制:
  - 个人微信 API 通过 iLink 实现，功能有限
  - 企业微信支持 Webhook，更稳定
